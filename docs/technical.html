Technical Challenges

The hard engineering problems in this product.

This page is only about technical challenges and current status.

Top 5 Technical Challenges

  1. Legal text to code rules. We must convert legal language into clear, testable rules without losing legal meaning.
  2. Evidence quality, not just citations. A random link is not enough. We need a source quality model compliance teams can trust.
  3. Deterministic outputs for audits. Same input + same policy version must always return the same result.
  4. Low-friction integration. If setup is hard or slows CI too much, teams will bypass the tool.
  5. Policy version control across regions. Country-specific overrides must stay consistent and traceable.

What Is Working Today

  • Typed schema layer for claims and policies.
  • Policy-driven validation engine.
  • CLI flow for generate/validate and JSON outputs.
  • CI-style pass/fail exit behavior.

What Is Not Solved Yet

  • Mature evidence quality scoring model.
  • Complete rule mapping by jurisdiction.
  • Compliance reviewer UX for sign-off workflows.
  • Pilot-proven reduction in legal incidents.

Where Legal Context Should Live

In versioned policy packs. Each rule should include jurisdiction, source URL, effective date, rule ID, and risk level. Every validation output should reference those IDs for replay and audits.

Country Rollout Approach

Start with one baseline rule pack first, then add country-level overrides. Do not launch multi-country logic before one rule pack is stable and tested.

Most Important Technical Questions For The Pitch

  1. What legal baseline pack is v1?
  2. How do we prove source quality in outputs?
  3. How do we guarantee deterministic reruns?
  4. How fast can a dev team integrate in CI?
  5. What metric proves technical value in a pilot?
